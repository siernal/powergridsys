# Архитектурный документ ВКР

**Тема:** Система учёта обслуживания электрических сетей с модулем прогнозирования отказов и планирования ремонтов на основе цифровой копии электросети.

**Кодовое имя:** PowerGrid Maintenance System
**Тип:** Демонстрационный MVP, работает на синтетических данных.

---

## 1. Цели и задачи

**Цель работы.** Разработать веб-систему класса EAM/CMMS со встроенным модулем предиктивной аналитики и упрощённой цифровой копией электросети, способную в условиях отсутствия реальных данных и без подключения к SCADA продемонстрировать:

- ведение реестра объектов электросетевого хозяйства;
- учёт осмотров, ремонтов, отказов;
- автоматический расчёт риска отказа на основе модели машинного обучения;
- автоматическое формирование плана ТОиР по приоритетам.

**Задачи:**

1. Спроектировать схему базы данных для предметной области.
2. Сгенерировать реалистичный синтетический датасет (объекты, история обслуживания, погода, телеметрия).
3. Построить упрощённую цифровую копию (digital twin) электросети в виде ориентированного графа.
4. Обучить baseline-модель прогнозирования отказов (Random Forest).
5. Реализовать алгоритм планирования ТОиР по риск-приоритетам.
6. Реализовать REST API и веб-интерфейс.
7. Упаковать систему в Docker для воспроизводимого запуска.

## 2. Архитектура

Трёхзвенная клиент-серверная архитектура:

```
┌────────────────┐    REST/JSON     ┌─────────────────────┐    SQL    ┌──────────────┐
│   Frontend     │ ───────────────► │       Backend       │ ────────► │  PostgreSQL  │
│  React 18 + TS │ ◄─────────────── │  FastAPI + ORM      │ ◄──────── │      15      │
│  Vite, Recharts│                  │  SQLAlchemy + JWT   │           └──────────────┘
└────────────────┘                  └────────┬────────────┘
                                             │
                          ┌──────────────────┴──────────────────┐
                          ▼                                     ▼
                ┌──────────────────┐                ┌──────────────────────┐
                │  Digital Twin    │                │  ML Predictor        │
                │  NetworkX-граф   │                │  scikit-learn RF     │
                │  health/risk     │                │  pickle, training set│
                └──────────────────┘                └──────────────────────┘
```

**Стек обоснован тем, что:**

- *FastAPI* — современный async-веб-фреймворк с автогенерируемым Swagger UI; идеален для дипломной защиты, т.к. позволяет демонстрировать API живьём.
- *SQLAlchemy 2.0* — индустриальный ORM, отделяет бизнес-логику от SQL.
- *PostgreSQL* — реляционная БД промышленного уровня; поддерживает JSON-поле для feature_snapshot.
- *NetworkX* — Python-библиотека графов; даёт алгоритмы поиска путей, потомков, циклов «из коробки».
- *scikit-learn Random Forest* — ансамблевая модель, устойчивая к шуму, не требует масштабирования, даёт `feature_importances_` для объяснения решения.
- *React + Vite* — быстрый и популярный стек, минимальный TTFB в dev-режиме.
- *Recharts* — декларативные графики на SVG, хорошо подходят для дашбордов.

## 3. Схема базы данных

| Таблица               | Назначение                                                       |
|------------------------|------------------------------------------------------------------|
| `users`               | Учётные записи (admin / engineer / inspector / viewer), bcrypt-пароли |
| `asset_types`         | Справочник типов оборудования: трансформатор, ВЛ, подстанция, кабель |
| `assets`              | Физические объекты сети: координаты, регион, класс напряжения, критичность |
| `inspections`         | Журнал осмотров, оценка состояния 1..5, дефекты                  |
| `repairs`             | История ремонтов (planned/emergency/capital), стоимость, бригада |
| `failure_events`      | Зафиксированные отказы: тип, тяжесть (minor/major/critical), даунтайм |
| `risk_scores`         | История расчётов риска: вероятность, уровень, JSON-snapshot признаков |
| `maintenance_plans`   | Сгенерированный план ТОиР: дата, приоритет, тип, оценка стоимости |
| `sensor_snapshots`    | Имитация телеметрии (load %, темп., вибрация) — каждые 6 ч за 30 дней |
| `weather_snapshots`   | Имитация погодных данных по регионам — суточные, год назад       |

Связи:

```
asset_types 1 ─── n assets
assets 1 ─── n inspections / repairs / failure_events / risk_scores
              / maintenance_plans / sensor_snapshots
users 1 ─── n inspections
weather_snapshots ←─ region (без жёсткого FK, многие-ко-многим через регион)
```

Полная SQL-схема генерируется автоматически из ORM (`Base.metadata.create_all`).
Альтернативно её можно получить как DDL: `python -c "from core.database import Base, engine; print('\n'.join(str(t) for t in Base.metadata.sorted_tables))"`.

## 4. Алгоритм генерации синтетических данных

Используются два независимых генератора.

### 4.1 Демонстрационный заполнитель БД (`backend/seed/seed_data.py`)

Формирует «исторический» снимок реальной сети:

- **120 объектов** со сбалансированными весами по типам (трансформаторы 25 %, ВЛ 38 %, подстанции 12 %, кабели 25 %).
- **Возраст** — `lognormal(μ=2.5, σ=0.7)`, обрезается до 55 лет; даёт правую асимметрию (медиана ≈ 12 лет, длинный хвост старого оборудования).
- **Координаты** имитируют сеть в Московской области (55.5° N ± 1.5, 37.0° E ± 2).
- **Осмотры** — `~age·0.8` штук на объект, оценка состояния по биномиальному распределению с пиком на 4.
- **Ремонты** — `~age/5` штук, тип взвешенно (planned 60 %, emergency 30 %, capital 10 %).
- **Отказы** — `Poisson(age/15 · criticality)`. Тяжесть взвешенно minor/major/critical (50/30/20). Downtime коррелирует с тяжестью.
- **Погода** — 365 суток × 5 регионов; температура `T = -10 + 25·sin(π·(month−3)/6) + N(0,4)`, шторм при `wind > 15 ∧ precip > 10`.
- **Телеметрия** — 30 дней × 4 точки/сутки, нагрузка `N(65, 12)` + сезонный сдвиг.

### 4.2 Обучающий датасет ML (`backend/services/ml_predictor.generate_training_data`)

6000 строк, признаки и метка `failure_label`:

```
score = 0.045·age
      + 0.28·failures_count
      − 0.18·min(repairs_count, 10)         # ремонты снижают риск
      + 0.55·criticality
      + 0.012·max(0, load − 70)              # перегрузка
      + 0.0005·days_since_maintenance
      + 0.0003·days_since_inspection
      + 0.20·is_storm
      + season_factor                         # +0.15 зимой, +0.08 летом
      + cat_bias                              # transformer 0.10, substation 0.15
      + N(0, 0.15)                            # шум
      − 3.2                                   # сдвиг ↦ ~25 % положительных
label = Bernoulli(σ(score))
```

Такой генератор сохраняет монотонные зависимости (старый объект с высокой нагрузкой и шторм → выше риск), но добавляет шум, чтобы модель не выучивала формулу один-в-один.

## 5. Алгоритм Digital Twin

Реализован как singleton класса `DigitalTwin` поверх `networkx.DiGraph`.

### 5.1 Построение графа

```
substation ──► transformer ──► line ──► cable
```

Каждому трансформатору приписывается случайная подстанция (детерминированно: `random.seed(1)`); каждой линии — случайный источник из {трансформаторы ∪ подстанции}; каждому кабелю — случайный источник из {трансформаторы ∪ линии}.

### 5.2 Расчёт состояния узла

```
health = 100
       − 0.9·age_years
       − 6.0·failures_count
       + 1.5·min(repairs_count, 8)
       − 0.25·max(0, load_percent − 70)
       − 8·min(days_since_maintenance / 365, 3)
       − 5  if is_storm
       − 4  if temperature ∉ [−20, 40]
       − (1 − health/100)·criticality·5      # критичные деградируют быстрее
health ← clip(health, 0, 100)

risk = sigmoid((50 − health) / 15)            # health=50 ↦ risk≈0.5
```

### 5.3 Каскадный анализ

Для узлов с `risk > 0.65` собирается `nx.descendants(graph, node)` — это объекты, которые останутся без питания при отказе данного узла. Используется при отображении «Critical chains» на дашборде.

## 6. ML-модель прогнозирования отказов

### 6.1 Конфигурация

```python
RandomForestClassifier(
    n_estimators=250,
    max_depth=8,
    min_samples_leaf=10,
    class_weight="balanced",
    random_state=42,
    n_jobs=-1,
)
```

### 6.2 Признаки

| Группа        | Признаки                                                                           |
|---------------|------------------------------------------------------------------------------------|
| Сам объект    | `category` (one-hot), `age_years`, `criticality`                                   |
| История       | `failures_count`, `repairs_count`, `days_since_maintenance`, `days_since_inspection` |
| Среда         | `month`, `is_storm`, `humidity`, `temperature`                                     |
| Эксплуатация  | `load_percent`                                                                     |

Итого 11 числовых + 4 one-hot = 15 признаков.

### 6.3 Метрики

На синтетике типично достигается `roc_auc ≈ 0.83–0.85`, `f1_macro ≈ 0.74`. Метрики сохраняются вместе с моделью (`pickle`) и доступны через `GET /api/risk/model-metrics`.

### 6.4 Классификация уровня

| Вероятность отказа | Уровень  |
|--------------------|----------|
| `< 0.30`           | low      |
| `0.30 ≤ p < 0.60`  | medium   |
| `≥ 0.60`           | high     |

### 6.5 Обоснование выбора

В тексте ВКР удобно сравнить с **Logistic Regression** как с бейзлайном:
- LR быстрее и интерпретируемее, даёт сопоставимый AUC;
- RF лучше на нелинейных взаимодействиях (например, `age × criticality`) и предоставляет `feature_importances_`, что позволяет визуализировать «топ-5 признаков» при каждом предсказании.

Для будущего расширения предусмотрено место под XGBoost / LightGBM (тот же интерфейс `FailurePredictor.predict`).

## 7. Алгоритм планирования ТОиР

Файл: `backend/services/maintenance_planner.py`.

```
priority(asset) = 0.50·risk_probability
                + 0.30·criticality
                + 0.20·min(days_since_maintenance / 365, 3) / 3
```

Сортировка убывающая. Для каждого объекта:

| Уровень риска    | Окно планирования | Тип работ              |
|------------------|-------------------|------------------------|
| `risk ≥ 0.60`    | 1–30 дней         | `emergency_inspection` |
| `0.30 ≤ risk < 0.60` | 15–90 дней    | `planned_maintenance`  |
| `risk < 0.30`    | 60–180 дней       | `routine_inspection`   |

Ограничение `MAX_PER_DAY = 5` — имитация ёмкости ремонтных бригад. Стоимость работ:

```
cost = base_cost · (0.8 + criticality · 0.5)
base_cost = 50 000 / 120 000 / 15 000 ₽
```

Каждый запуск генератора удаляет старые auto-generated планы со статусом `scheduled` и пересоздаёт их заново.

## 10. Безопасность

- Аутентификация — JWT (HS256), срок жизни 8 часов.
- Хеширование паролей — bcrypt через `passlib`.
- Роли (`admin`/`engineer`/`inspector`/`viewer`) хранятся в `users.role`, проверка ролей в API оставлена точкой расширения для дальнейшей работы.
- CORS открыт для `http://localhost:5173` (dev-режим Vite).

---

## 11. Деплой

Один файл `docker-compose.yml` поднимает три сервиса:

1. `db` — PostgreSQL 15 с healthcheck;
2. `backend` — FastAPI на uvicorn (auto-reload включён, на старте: создаёт таблицы → запускает seed → обучает ML, если не было обучения → строит digital twin);
3. `frontend` — Vite dev-server на `:5173`.

Время холодного старта на типичном ноутбуке: ~30 секунд (postgres + миграции + seed + обучение модели).

## 12. Что демонстрируется на защите

1. Открыть `http://localhost:5173`, показать дашборд с метриками: 120 объектов, X в зоне высокого риска.
2. Перейти в реестр, отфильтровать по типу/региону.

7. Открыть `/docs` — показать Swagger.
